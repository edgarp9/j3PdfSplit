# Codex Agent Instructions (Python/Tkinter 프로젝트)

이 문서는 Codex가 Python/Tkinter 소형 프로젝트에서 일관되게 작업하기 위한 기준입니다.  
항상 한국어로 답변합니다. 코드/커밋 메시지/주석은 필요 시 영어를 섞어도 됩니다.

## 1) 핵심 원칙

1. 작은 프로젝트라도 유지보수 가능한 구조를 우선합니다.
2. UI와 로직과 외부 I/O를 한 함수에 섞지 않습니다.
3. Tkinter 메인 루프를 블로킹하지 않습니다.
4. 예외를 숨기지 말고, 내부 원인과 사용자 메시지를 분리합니다.
5. 새 의존성은 신중히 추가합니다.
   - 표준 라이브러리 우선
   - 기존 의존성 재사용 우선
6. 소스 수정이 있으면 최소 검증까지 완료합니다.

## 2) 권장 구조

소형 프로젝트에서는 `ui -> app -> infra` 단방향 의존만 유지하면 충분합니다.

- `ui`
  - Tkinter 위젯, 이벤트 바인딩, 화면 갱신
- `app`
  - 상태 관리, 작업 흐름, 입력 검증, 백그라운드 작업 연결
- `infra`
  - 파일, 네트워크, OS 연동 등 외부 I/O
- `domain`
  - 순수 로직이 커질 때만 별도 분리

중요한 것은 폴더 수가 아니라, 역할이 섞이지 않도록 유지하는 것입니다.

예시:

```text
main.py
ui.py
app.py
infra.py
```

필요 시 확장:

```text
ui/
app/
infra/
domain/   # 필요할 때만 추가
```

## 3) 문서주도 DDD

- 기본 문서 파일은 `docs/domain.md`로 둡니다.
- 기능 변경 전 요구사항, 용어, 규칙을 먼저 문서에 짧게 정리합니다.
- 도메인 용어는 코드 이름과 문서에서 일관되게 사용합니다.
- 유스케이스 흐름은 `app`에 두고, 핵심 규칙은 가능하면 `domain` 또는 순수 함수로 분리합니다.
- 문서와 코드가 달라지면 코드 수정과 함께 문서를 갱신합니다.
- 작은 프로젝트에서는 과한 계층보다 명확한 책임 분리를 우선합니다.

## 4) Tkinter 규칙

- UI 코드에서 파일 I/O, 네트워크, 긴 연산을 직접 처리하지 않습니다.
- 위젯 조작은 메인 스레드에서만 수행합니다.
- 백그라운드 작업 결과는 `queue + after` 또는 `after(0, ...)`로 UI에 반영합니다.
- `PhotoImage` 등 UI 리소스는 참조를 유지합니다.
- 대량 갱신은 가능하면 부분 업데이트를 우선합니다.
- 긴 작업은 가능하면 취소 가능하거나, 완료 시점에 무시 가능한 상태 플래그를 둡니다.
- 창 종료 시 진행 중인 작업, 타이머, 큐 소비 루프가 안전하게 정리되도록 합니다.
- 팝업창/대화상자는 모달 방식으로 띄울 것.
- 팝업창/대화상자는 메인 윈도우 기준 중앙에 배치하기.


## 5) 에러 처리 / 로깅

- `except: pass` 금지
- 내부에는 예외 원인과 로그를 남기고, UI에는 이해 가능한 메시지만 보여줍니다.
- 로깅은 기본적으로 `logging`을 사용합니다.

## 6) 플랫폼 분리

- Windows/Linux 전용 처리는 가능한 한 `infra` 또는 전용 헬퍼에 둡니다.
- 공용 로직에서 플랫폼 분기를 남발하지 않습니다.
- 다른 OS에서 import 단계부터 깨지지 않도록 가드된 import 또는 지연 import를 사용합니다.

## 7) 작업 방식

변경 전:
- 기존 구조, 파일 배치, 네이밍, 호출 흐름을 먼저 확인합니다.
- 이미 있는 유틸/함수를 우선 재사용합니다.

구현 시:
- 복잡한 로직은 함수로 분리합니다.
- 매직 넘버/설정값은 상수화합니다.
- 공개 함수/클래스에는 필요 시 docstring을 추가합니다.
- 한 번에 큰 변경보다 작은 단계로 나누어 수정합니다.
- 추측으로 구조를 바꾸지 말고, 확인된 문제와 목적에 근거해 수정합니다.

변경 후:
- 변경 파일
- 핵심 수정 내용
- 검증 결과
- 남은 리스크
를 함께 보고합니다.

## 8) 검증 기준

소스 수정이 있으면 아래 중 가능한 항목을 최소 1개 이상 수행합니다.

- `python -m compileall .`
- `python -m pytest`
- `python -m unittest`
- `ruff check .`

가능하면 앱 실행 스모크도 확인합니다.

## 9) 금지 사항

- 백그라운드 스레드에서 Tkinter 위젯 직접 조작
- 메인 루프 블로킹
- `except: pass` 남용
- 기존 패턴 무시한 중복 구현
- 플랫폼 전용 코드를 UI/공용 로직에 무분별하게 혼합
- `.my` 폴더 접근 또는 분석

추가 제외 경로/패턴:

`dist/`, `.git/`, `.my/`, `build/`, `log/`, `__pycache__/`, `*.pyd`, `*.pyc`, `*.pyo`, `*$py.class`, `.venv/`, `venv/`, `env/`, `ENV/`, `site-packages/`, `.ruff_cache/`, `.pytest_cache/`, `.mypy_cache/`, `.pyre/`, `.hypothesis/`, `.tox/`, `.nox/`, `.eggs/`, `*.egg-info/`, `pip-wheel-metadata/`, `.coverage`, `.coverage.*`, `coverage.xml`, `htmlcov/`, `*.log`, `*.tmp`, `*.bak`, `.vscode/`, `.idea/`, `.DS_Store`, `Thumbs.db`, `Desktop.ini`, `*~`, `*.swp`, `*.swo`, `.nfs*`, `.fuse_hidden*`, `.directory`, `.Trash-*`, `.xsession-errors*`, `data/`

기본적으로 위 제외 경로/패턴과 일치하는 파일·디렉터리 및 그 하위 항목은 읽기, 검색, 후보 탐색, evidence 인용, 수정 후보 산정, 수정 지시문 생성 대상에서 모두 제외합니다. 또한 해당 경로 내부에서는 파일 생성, 수정, 삭제, 이동, 이름 변경, 포맷팅을 수행하지 않습니다.

단, 사용자가 위 제외 경로/패턴에 해당하는 대상에 대해 명시적으로 작업을 지시한 경우에는, 해당 지시 범위 안에서만 읽기, 검색, 생성, 수정, 삭제, 이동, 이름 변경, 포맷팅을 수행할 수 있습니다.

## 10) 빠른 체크리스트

- [ ] UI / 로직 / I/O가 섞이지 않았는가
- [ ] 문서의 요구사항, 도메인 용어, 코드 이름이 일치하는가
- [ ] 메인 루프를 블로킹하지 않는가
- [ ] 위젯 조작이 메인 스레드에서만 일어나는가
- [ ] 예외를 숨기지 않았는가
- [ ] 최소 검증을 수행했는가
- [ ] 커밋 메시지에 요약과 수정사항이 포함됐는가

## 과도한 설계 피하기

깔끔하고 읽기 쉽고 유지보수하기 쉬운 코드를 작성하되, 불필요한 설계는 피하십시오.

중복, 실제 변경 압박, 테스트 용이성 또는 기존 프로젝트 패턴으로 정당화되기 전까지는 단순하고 구체적인 구현을 우선하십시오.

미래에 필요할 수도 있다는 이유만으로 인터페이스, 팩토리, 전략 패턴, 제네릭 레이어, 래퍼, 설정 기반 구조를 도입하지 마십시오.

모든 추상화는 현재 코드베이스 안에서 구체적인 이유가 있어야 하며, 추가할 경우 최소한으로 유지하고 주변 코드와 일관되게 작성하십시오.
